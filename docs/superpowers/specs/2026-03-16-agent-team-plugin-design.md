# Agent Team Plugin — Design Specification

A Claude Code plugin that dynamically assembles and orchestrates a team of specialized AI agents to build Databricks applications end-to-end, from data ingestion to deployed app.

## Overview

Two slash commands power the system:

- **`/create-team <prd-path>`** — Analyzes a PRD, selects/generates agent personas, designs phase structure, defines inter-agent contracts, and writes everything to `.agent-team/` for human review.
- **`/start-team [flags]`** — Dispatches a PM orchestrator agent that executes the phased plan, coordinates agents in parallel worktrees, runs progressive QA gates, and auto-recovers from interruptions.

### Validation Target

The system will be validated by building a **Q&A chatbot over documentation** — ingest a docs corpus, build a RAG pipeline with Vector Search, and serve it through a simple chat UI as a Databricks App. This exercises: Data Engineer (ingestion/chunking), GenAI Architect (RAG pipeline/embeddings/Vector Search), App Developer (chat UI), and Deploy Engineer — without requiring ML model training, keeping the test surface focused on orchestration quality.

## Architecture

### Delivery Mechanism

Claude Code plugin with:
- Two registered slash commands (`/create-team`, `/start-team`)
- One registered agent (`pm-orchestrator`)
- Curated agent prompt templates in `templates/core/`
- Meta-templates for generating dynamic specialist agents in `templates/meta/`

### Dispatch Model

**Prompt-Template Engine** — All agents (curated and dynamic) are dispatched as `general-purpose` subagents via the `Agent()` tool. Each agent's full prompt is constructed from its definition file, with template variables resolved at dispatch time by the PM orchestrator. The `model` parameter controls tier assignment per agent.

### Agent Isolation

Every agent runs in an isolated **git worktree** (`isolation: "worktree"` on the Agent tool). Agents write to non-overlapping directory paths within the DAB structure. The PM merges worktree branches into main after each phase.

### Coordination Model

**Layered — PM coordinates phases, agents self-coordinate within phases.**

- PM agent (Opus) breaks work into phases and dispatches agents per phase
- Within a phase, parallel agents work independently against shared contracts
- Between phases, PM reviews results, merges worktrees, and runs QA
- Human gets pulled in at phase boundaries and for unresolvable blockers

## Agent Template Format

Each agent is defined as a Markdown file with YAML frontmatter:

```yaml
---
name: data-engineer
display_name: Data Engineer
model: sonnet                    # sonnet | opus | haiku
phase: [1]                       # Which phases this agent participates in
parallel_group: data             # Agents in same group run concurrently

skills:                          # Claude Code skills this agent should invoke
  - spark-declarative-pipelines
  - databricks-unity-catalog
  - databricks-query
  - asset-bundles

mcp_tools:                       # MCP tools to emphasize in prompt
  - databricks-mcp:execute_sql
  - databricks-mcp:get_table_details
  - databricks-mcp:manage_uc_objects

inputs:                          # What this agent needs before starting
  - contract: domain-playbook
    from: docs-domain-sme
  - contract: data-requirements
    from: phase-0-planning

outputs:                         # What this agent produces
  - contract: data-to-ml
    artifacts:
      - src/ingestion/**
      - src/transformations/**
      - resources/pipelines.yml

constraints:                     # Guardrails for this agent
  - Write code to src/ and resources/ only
  - Do not provision infrastructure directly
  - All tables in Unity Catalog under project schema
  - Follow DAB project structure
---

# Role
You are a Senior Databricks Data Engineer on a cross-functional
team building {{project_description}}.

# Objective
{{objective}}

# Context
{{phase_context}}
{{domain_playbook}}
{{contract_inputs}}

# Output Requirements
- Write pipeline code to src/ingestion/ and src/transformations/
- Define pipeline resources in resources/pipelines.yml
- Produce tables matching your output contracts
- Write unit tests to tests/data_engineering/

# Status Protocol
When finished, write status to .agent-team/status/{{agent_name}}.yaml:
  status: DONE | DONE_WITH_CONCERNS | NEEDS_CONTEXT | BLOCKED
  artifacts: [list of files created/modified]
  concerns: [if any]
  blockers: [if any]
```

### Model Tier Assignment

| Tier | Role | Agents |
|------|------|--------|
| **Opus** | Complex/strategic | PM Orchestrator, GenAI Architect, phase planning, cross-agent conflict resolution |
| **Sonnet** | Implementation | Data Engineer, Data Scientist, App Developer, Deploy Engineer, QA Engineer |
| **Haiku** | Lightweight | Domain SME (web research), contract validators, status summarizers, file search/review |

### Dynamic Agent Generation

For capabilities not covered by curated templates (e.g., `domain:documentation`, `domain:retail`), `/create-team` uses meta-templates to generate novel agent definitions:

1. Opus reads the PRD domain and the meta-template (`domain-sme-generator.yaml` or `specialist-generator.yaml`)
2. Generates a full agent `.md` file with frontmatter, prompt, skills, and tools
3. The Domain SME is assigned `WebSearch` and `WebFetch` tools to research the domain
4. SME runs in Phase 0 and produces a `domain-playbook.md` — a broadcast contract consumed by all downstream agents

### Meta-Template Format

Meta-templates instruct Opus how to generate a novel agent definition:

```yaml
# templates/meta/domain-sme-generator.yaml
type: generator
output_format: agent-definition-md    # Produces a full agent .md file

generator_prompt: |
  Given the PRD domain "{{domain}}" and project "{{project_description}}",
  generate a Domain Subject Matter Expert agent definition.

  The SME must:
  1. Have deep knowledge of {{domain}} industry workflows
  2. Research current best practices using WebSearch/WebFetch
  3. Produce a domain-playbook.md covering:
     - Industry-standard workflows relevant to the project
     - Key data sources and their typical schemas
     - Success metrics and KPIs
     - Regulatory or compliance considerations
     - Common pitfalls and how to avoid them
  4. Be assigned model: haiku (research-focused, not code-heavy)
  5. Run in Phase 0 before all other agents
  6. Output a broadcast contract consumed by all agents

  Generate the agent definition using the standard frontmatter format
  (name, display_name, model, phase, skills, mcp_tools, inputs,
  outputs, constraints) followed by the full role prompt.

defaults:
  model: haiku
  phase: [0]
  skills: []
  mcp_tools: []
```

### Contract Schema Format

`lib/contract-schema.yaml` defines the valid structure for all contracts:

```yaml
# Required fields
name: string                    # Unique contract identifier
producer: string                # Agent name that produces the output
consumer: string | "broadcast"  # Agent name or "broadcast" for all agents

# Optional: table definitions
tables:
  - name: string                # Table name
    catalog: string             # UC catalog (supports {{template_vars}})
    schema: string              # UC schema (supports {{template_vars}})
    description: string
    columns:
      - name: string
        type: string            # Spark SQL type
        required: boolean
    min_rows: integer           # Minimum expected row count

# Optional: file artifact definitions
artifacts:
  - path: string                # Glob or directory path
    description: string

# Optional: validation rules
validation:
  - type: schema_match | artifact_exists | code_references | sql_check
    description: string
    query: string               # For sql_check type
    expect: string              # For sql_check type
```

### Dispatch Mechanism

The PM orchestrator reads an agent definition, resolves all `{{template_variables}}`, and dispatches via:

```
Agent(
  description: "Data Engineer - Phase 1 ingestion"
  model: "sonnet"
  prompt: "{resolved agent definition with all template vars filled}"
  isolation: "worktree"
  run_in_background: true   # For parallel agents
)
```

For parallel agents in the same phase, PM dispatches multiple `Agent()` calls in a single message.

## Inter-Agent Contracts

Contracts define what a producer agent creates and what a consumer agent expects:

```yaml
name: data-to-genai
producer: data-engineer
consumer: genai-architect

tables:
  - name: doc_chunks
    catalog: "{{project_catalog}}"
    schema: "{{project_schema}}"
    description: Chunked and cleaned documentation for embedding
    columns:
      - name: chunk_id
        type: STRING
        required: true
      - name: doc_source
        type: STRING
        required: true
      - name: chunk_text
        type: STRING
        required: true
      - name: chunk_index
        type: INT
        required: true
      - name: metadata
        type: STRING
        required: false
    min_rows: 50

artifacts:
  - path: src/ingestion/
    description: Document ingestion and chunking pipelines
  - path: src/transformations/
    description: Text cleaning and chunk processing
  - path: resources/pipelines.yml
    description: SDP pipeline definitions

validation:
  - type: schema_match
    description: Consumer can read all required columns
  - type: artifact_exists
    description: All listed artifacts exist in project
  - type: sql_check
    query: "SELECT COUNT(*) FROM {{catalog}}.{{schema}}.doc_chunks"
    expect: "> 0"
```

### Contract Chain (Q&A Chatbot)

```
Docs Domain SME ──domain-playbook──→ ALL AGENTS (broadcast)

Data Engineer ──data-to-genai──→ GenAI Architect ──genai-to-app──→ App Developer ──app-to-deploy──→ Deploy Engineer
```

### Contract Validation

QA validates contracts at phase boundaries using 4 checks:

1. **artifact_exists** — all listed files/directories exist in project
2. **schema_match** — producer's code produces columns matching contract schema
3. **code_references** — consumer's code references the contracted tables
4. **custom_checks** — SQL or script checks defined in the contract

## /create-team Command

### Signature

```
/create-team <prd-path>
/create-team docs/qa-chatbot-prd.md
/create-team https://docs.google.com/document/d/abc123
```

### Execution Pipeline (runs on Opus)

1. **Read & parse PRD** — local file or Google Doc via skill
2. **Identify capabilities needed** — map PRD requirements to capability tags (data-ingestion, genai-rag, vector-search, web-app, domain:documentation, etc.)
3. **Select curated agents** — match capability tags to templates in `templates/core/`
4. **Generate dynamic specialists** — for uncovered capabilities, use meta-templates to generate novel agent definitions
5. **Design phase structure** — analyze agent dependencies, group into phases with parallel groups
6. **Define contracts** — for each producer→consumer edge, generate a contract YAML with schemas, artifact paths, and validation rules
7. **Write .agent-team/ directory** — all agent definitions, phase configs, contracts, and initialized progress.yaml
8. **Present team summary** — display roster, phase plan, contract chain in terminal. Instruct user to review/edit before running `/start-team`

### Phase Structure Algorithm

- Agents with no upstream dependencies → earliest possible phase
- Agents consuming another's output → phase after producer completes
- Agents with no data dependency on each other → same phase, same parallel group
- Integration agent always comes after all producers
- Deploy always last

## /start-team Command

### Signature

```
/start-team                      # Run all phases (or auto-resume from checkpoint)
/start-team --phase 2            # Override: restart from phase 2
/start-team --agent data-engineer # Re-run a single agent
/start-team --dry-run            # Show execution plan without dispatching
```

#### `--agent` Flag Semantics

When re-running a single agent:
1. Load the agent's phase context from `progress.yaml` (uses the original phase)
2. Dispatch the agent in a fresh worktree with the current (possibly edited) agent definition
3. On completion, merge the worktree branch into main
4. Re-run the QA gate for that agent's phase to validate the fix
5. Update `progress.yaml` with the new agent status

### Startup Sequence

1. **Validate .agent-team/** — verify manifest, agent definitions, phases, contracts all exist and parse
2. **Initialize git repo** — if not already initialized, `git init` + initial commit (enables worktree isolation)
3. **Scaffold DAB project** — generate base DAB structure if not present
4. **Check checkpoint state** — read `progress.yaml`, determine resume point
5. **Dispatch PM orchestrator** — launch PM agent (Opus) with full manifest, phase plan, and contract chain

### PM Orchestrator Runtime Loop

For each phase:

1. **Read phase config** — load `phases/phase-N.yaml`, identify agents and parallel groups
2. **Resolve template variables** — inject phase_context, domain_playbook, contract_inputs into each agent's prompt
3. **Dispatch parallel agents** — for each parallel group, dispatch all agents with `isolation: "worktree"` and `run_in_background: true`
4. **Await completion** — agents return with status files
5. **Handle statuses:**
   - `DONE` → proceed to merge
   - `DONE_WITH_CONCERNS` → review concerns, decide if blocking
   - `NEEDS_CONTEXT` → provide context, re-dispatch
   - `BLOCKED` → attempt unblock or escalate to human
6. **Merge worktrees** — integrate agent branches into main; resolve trivial conflicts, escalate semantic conflicts
7. **QA gate** — dispatch QA agent with progressive scope; PASS → advance, FAIL → fix loop (max 3 attempts), 3x FAIL → escalate
8. **Update progress** — write checkpoint, commit to git

### Human Touchpoints

| Trigger | PM Shows You | Your Options |
|---------|-------------|--------------|
| Agent BLOCKED (unresolvable) | Blocker details, what was attempted | Provide guidance / skip / abort |
| Semantic merge conflict | Both sides, which agents produced them | Choose resolution / manual fix |
| QA failed 3x | Failure details, fix attempts | Manual fix / skip QA / adjust contract |
| Phase boundary | Phase summary, test results, next preview | Continue / pause / adjust plan |
| All phases complete | Final report, all artifacts, deploy status | Deploy / review / iterate |

## Checkpoint & Auto-Recovery System

### Checkpoint Model

`progress.yaml` tracks 7 discrete steps per phase plus per-agent status:

```yaml
project: qa-chatbot
started_at: 2026-03-16T10:30:00Z
last_updated: 2026-03-16T10:47:23Z

current_state:
  phase: 2
  step: dispatch_agents
  status: in_progress

phases:
  0:
    status: completed
    started_at: 2026-03-16T10:30:00Z
    completed_at: 2026-03-16T10:35:12Z
    steps:
      read_phase_config:    completed
      resolve_contracts:    completed
      dispatch_agents:      completed
      await_agents:         completed
      merge_worktrees:      completed
      qa_gate:              completed
      update_progress:      completed
    agents:
      docs-domain-sme:
        status: DONE
        worktree_branch: agent/docs-domain-sme
        merged: true
        artifacts: [.agent-team/artifacts/domain-playbook.md]
  1:
    status: completed
    started_at: 2026-03-16T10:35:15Z
    completed_at: 2026-03-16T10:42:30Z
    steps:
      read_phase_config:    completed
      resolve_contracts:    completed
      dispatch_agents:      completed
      await_agents:         completed
      merge_worktrees:      completed
      qa_gate:              completed
      update_progress:      completed
    agents:
      data-engineer:
        status: DONE
        worktree_branch: agent/data-engineer
        merged: true
        artifacts: [src/ingestion/, src/transformations/, resources/pipelines.yml]
  2:
    status: in_progress
    started_at: 2026-03-16T10:42:33Z
    steps:
      read_phase_config:    completed
      resolve_contracts:    completed
      dispatch_agents:      in_progress
      await_agents:         pending
      merge_worktrees:      pending
      qa_gate:              pending
      update_progress:      pending
    agents:
      genai-architect:
        status: dispatched
        worktree_branch: agent/genai-architect
        merged: false
      app-developer:
        status: pending
        worktree_branch: null
        merged: false

escalations: []
qa_attempts:
  0: { attempts: 1, passed: true }
```

### Checkpoint Write Protocol

1. Before starting a step: write `in_progress` to progress.yaml
2. As agents are dispatched/complete: update per-agent status
3. After step completes: write `completed` + `git commit` progress.yaml

Progress is committed to git after every step completion, ensuring `/start-team` can always recover from the last committed state.

### Auto-Recovery Logic

| Interrupted At | Recovery Action |
|----------------|-----------------|
| `read_phase_config` | Re-read config (idempotent) |
| `resolve_contracts` | Re-resolve (idempotent, upstream unchanged) |
| `dispatch_agents` | Check worktree branches — dispatched agents with commits are treated as completed; undispatched agents are dispatched now |
| `await_agents` | Read status files from returned agents; re-dispatch agents with no status file |
| `merge_worktrees` | Check git log for already-merged branches; merge only unmerged |
| `qa_gate` | Re-run QA from scratch (stateless, safe to re-run) |
| `update_progress` | Write progress and commit (idempotent) |

### Recovery UX

```
$ /start-team

⟳ Resuming from checkpoint...
  Phase 0 (Planning):        ✓ completed
  Phase 1 (Data Ingestion):  ✓ completed
  Phase 2 (RAG + App):       ▸ interrupted at dispatch_agents
    genai-architect:         ✓ completed (branch exists with commits)
    app-developer:           ✗ not dispatched — dispatching now
  Phase 3 (Integration):     ○ pending
  Phase 4 (Deploy):          ○ pending

Resuming Phase 2 — dispatching app-developer agent...
```

## Progressive QA

QA scope intensifies by phase:

| Phase | QA Scope | Model |
|-------|----------|-------|
| Phase 0 | PM reviews domain playbook completeness | Haiku |
| Phase 1 | Code quality (lint, types, unit tests) + contract schema validation | Sonnet |
| Phase 2 | Phase 1 checks + integration contract validation + API structure | Sonnet |
| Phase 3 | All previous + E2E test generation + `databricks bundle validate` + security review | Sonnet |
| Phase 4 | Deployed validation: pipeline executes, model serves, app responds + final report | Sonnet |

## Conflict Resolution & Escalation

| Scenario | PM Action |
|----------|-----------|
| Merge conflict (trivial) | PM resolves (different sections of same file) |
| Merge conflict (semantic) | Escalate to human with both sides and context |
| Agent BLOCKED | Provide more context + re-dispatch. If still blocked, escalate to human |
| Contract mismatch | QA catches at phase boundary. PM re-dispatches producer with fix instructions |
| QA gate failure | Dispatch fix agent, re-run QA. After 3 failures, escalate to human |
| Human escalation | PM writes context to `.agent-team/status/escalation.md`, returns control to main session |

## Q&A Chatbot — Concrete Phase Plan

### Phase 0: Planning
- **Agents:** Docs Domain SME (haiku)
- **Produces:** domain-playbook.md (documentation Q&A best practices, chunking strategies, evaluation approaches), data-requirements.yaml, success-criteria.yaml
- **QA:** PM reviews playbook completeness

### Phase 1: Data Ingestion
- **Agents:** Data Engineer (sonnet) — solo
- **Produces:** src/ingestion/ (doc loading, text extraction), src/transformations/ (chunking, cleaning), resources/pipelines.yml
- **QA:** Code quality + data-to-genai contract validation (doc_chunks table schema)

### Phase 2: RAG + App
- **Agents:** GenAI Architect (opus), App Developer (sonnet) — parallel group "app"
- **Produces:** src/genai/ (embedding pipeline, Vector Search index, retrieval chain, prompt templates), src/app/backend/ (FastAPI), src/app/frontend/ (chat UI), resources/serving.yml
- **Note:** Both agents code against the `genai-to-app` contract definition (endpoint shape, request/response schema) — not each other's live artifacts. Phase 3 wires them together.
- **QA:** Code quality + genai-to-app contract validation

### Phase 3: Integration
- **Agents:** App Developer (sonnet) — solo
- **Produces:** Wired chat app connecting to RAG endpoint, tests/e2e/
- **QA:** E2E test suite + full contract chain validation + `databricks bundle validate`

### Phase 4: Deploy & Validate
- **Agents:** Deploy Engineer (sonnet), then QA Engineer (sonnet) — sequential
- **Actions:** Finalize DAB config, `databricks bundle deploy --target dev`, deployed smoke tests
- **QA:** Pipeline executes, RAG endpoint responds, chat UI loads, final project report
